2. SDK JS (recommandé)
Ajoute le script, initialise avec ta clé, puis envoie des events où tu veux dans ton app.
<script src="https://devconnect-dev.github.io/Nexus/nexus-sdk.js"></script>
<script>
  Nexus.init('nx_live_VOTRE_CLE');

  // Quand un évènement pertinent se produit dans ton app :
  Nexus.track('login_failed', {
    user_ref: 'user-42',
    severity: 'medium',
    details: { ip: '1.2.3.4' }
  });
</script>
3. Ou en HTTP direct (backend, curl...)
Aucun SDK requis : un simple POST JSON suffit, idéal depuis un serveur.
curl -X POST https://ekezdtageniidwjlgyys.supabase.co/functions/v1/nexus-ingest \
  -H "X-Nexus-Key: nx_live_VOTRE_CLE" \
  -H "Content-Type: application/json" \
  -d '{"event_type":"login_failed","user_ref":"user-42","severity":"medium","details":{"ip":"1.2.3.4"}}'
4. Référence rapide
event_type — chaîne libre définie par toi (ex : login_failed, signup, checkout_blocked, spam_detected). Reste cohérent d'un event à l'autre pour que les règles automatiques fonctionnent.
user_ref — optionnel, l'identifiant de l'utilisateur concerné dans ton système (peut être n'importe quelle chaîne).
severity — optionnel parmi info, medium, high, critical. Si omis, Nexus l'infère depuis le type d'event.
details — objet JSON libre (contexte, IP, payload...), utile pour l'investigation dans les Logs.

Chaque appel décrémente ton quota mensuel (voir plan ci-dessus). Au-delà, l'API répond 429 et l'event n'est pas enregistré.
Roadmap
Pistes à l'étude pour l'évolution de Nexus — rien n'est encore engagé, ceci sert de tableau de bord d'idées
● Candidat proche terme ● Gros chantier (R&D nécessaire) ● Concept exploratoire
Nexus devient un réseau, pas juste un outil solo
Threat intel partagée entre projetsGros chantier
Les patterns d'attaque détectés sur l'ensemble des projets Nexus sont anonymisés et mutualisés dans une base commune — un nouveau site bénéficie directement de ce que les autres ont déjà appris. Un peu le principe d'un "Cloudflare pour petites communautés / SaaS".
Score de réputation IP / device cross-projetGros chantier
Un attaquant repéré sur un site Nexus devient automatiquement suspect sur les autres projets du réseau.
IA qui bosse à ta place
Auto-génération de règles par LLMCandidat proche
Nexus analyse 7 jours de logs, repère les patterns anormaux tout seul et propose des règles prêtes à activer ("j'ai vu 40 tentatives similaires, tu veux que je bloque ce pattern ?") plutôt que de tout créer à la main.
Copilote incidentCandidat proche
Quand un incident critique se déclenche, un résumé généré explique ce qui s'est passé, la gravité et l'action recommandée — plus lisible qu'un log brut.
Ce qui se montre / se partage
Badge "Protégé par Nexus" embeddableCandidat proche
Badge public à intégrer sur le site du client, avec mini dashboard live (events bloqués ce mois) — effet marketing gratuit à chaque install.
Status page publique par projetCandidat proche
Type statuspage.io mais pour la sécu : "12 tentatives bloquées aujourd'hui, 0 incident actif" — de la transparence qui rassure les utilisateurs du client.
Mode "war room" temps réelConcept
Vue plein écran façon SOC de film : logs qui défilent, carte du monde avec les IP qui tapent. Pas indispensable, mais très efficace en démo ou en pitch.
Direction produit plus radicale
Firewall applicatif completGros chantier
Nexus passe de la détection/alerte à un vrai proxy devant l'API du client, qui bloque avant même que la requête touche son backend.
Marketplace de règles communautairesGros chantier
Les devs publient et partagent leurs configs de règles par stack (Discord bot, SaaS, forum...) — Nexus devient une plateforme plutôt qu'un produit fermé.
Fingerprinting comportementalConcept
Au-delà du comptage d'events : analyser le rythme de frappe / navigation d'un compte pour distinguer un bot d'un humain, même sans event suspect explicite.
Honeypot automatiqueGros chantier
Nexus génère de faux endpoints / faux boutons admin invisibles pour un humain normal — tout ce qui les touche est forcément un scraper ou un bot, ban instantané sans faux positif possible.
Corrélation temporelle multi-comptesConcept
Détecter des comptes qui agissent en synchronisation parfaite (bot farm) même avec des IP différentes, uniquement sur le pattern de timing.
Gamification du côté défense
Mode "capture the flag" publicConcept
Un environnement de test où n'importe qui peut essayer d'attaquer un faux projet Nexus pour voir le moteur réagir en direct — génère du contenu, prouve que ça marche, et recrute des hackers curieux comme early testers.
Classement "meilleurs admins Nexus"Candidat proche
Score de sécu par projet basé sur la config des règles et le temps de réponse aux incidents — pousse les clients à mieux configurer, et donne un levier marketing ("les projets Nexus les mieux notés").
Extension du domaine
Nexus pour Discord / Telegram natifGros chantier
Bot plug-and-play qui utilise le même moteur, zéro-code pour les serveurs communautaires sans dev — un marché à part entière à côté du DevConnect actuel.
Mode "red team as a service"Concept
Nexus attaque volontairement le site du client (avec permission) pour tester ses défenses avant qu'un vrai attaquant le fasse — un pentest automatisé en abonnement mensuel.
Le plus délire
Prédiction d'attaqueConcept
À partir des patterns globaux (la threat intel partagée), Nexus prévient avant qu'une attaque n'arrive sur un projet spécifique : "vague de credential stuffing détectée sur 3 autres projets similaires au tien, active le mode strict".
Kill switch physiqueConcept
Intégration webhook vers un objet réel (lampe qui clignote rouge, notif Discord perso) quand un incident critique se déclenche — pour voir que ça chauffe même loin du PC.
